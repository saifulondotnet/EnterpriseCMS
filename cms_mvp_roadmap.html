High risks
Block builder scope creep — DnD editors are deceptively complex. Fix scope to 8 core blocks for MVP; resist adding more until launch.
Auth shortcuts — Skipping MFA or SSO is acceptable in MVP, but JWT refresh token rotation and login throttling are non-negotiable from day one.
GDPR non-compliance — Even in MVP, the consent banner and data erasure flow must ship. Non-compliance is a legal risk regardless of project phase.
Medium risks
Theme engine flexibility — Build the theme contract (slots, hooks) correctly in Sprint 3; retrofitting it after the block builder exists is costly.
Performance debt — Don't defer all performance work to Sprint 9. Add output caching headers and image lazy loading in Phase 2 alongside features.
Team capacity — 12 sprints is tight for a solo developer. With a 2-person team it's achievable; with 3+ it allows parallel streams.
Key decisions to make now
DB strategy — Single-schema now; design tenant_id column in every table from day one so multi-tenancy in Phase 2 is a migration not a rewrite.
Block storage format — Store blocks as structured JSON (not HTML) so the headless API in Phase 2 can serve blocks to any front-end.
Plugin hook points — Add the event/hook infrastructure in Sprint 1 even without plugins; retrofitting hook points into existing services is painful.
